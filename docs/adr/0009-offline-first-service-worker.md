# App 离线可用：Service Worker 秒开优先 + 逐件超时降级

日期：2026-07-08。背景：实际公园无信号场景中，App 只剩浏览器标签的视觉快照，点击每日规划无任何反应——GitHub Pages 缓存 `max-age=600` 过期后断网重载整页失败。行程数据本就全部内联在 index.html 里，离线看不了纯属加载层问题。

## 决策

1. **加 Service Worker（sw.js），同源资源全部缓存**。这是仓库里第一个常驻后台脚本——一旦装进用户浏览器，坏版本很难赶走，故凡改 sw.js 必须极其克制、必须先本地验证"杀服务器后重载仍可用"。
2. **秒开优先（stale-while-revalidate）**：永远先出缓存版（断网/弱网 0 毫秒可用），后台静默拉新，下次进入生效。取舍：部署后用户要第二次打开才见新版——内容节奏是"出发前改好、路上看"，秒开压倒即时性。放弃 network-first：弱网（请求吊死而非干脆断开）下每次打开都要干等超时，恰是最痛场景。
3. **不做离线地图瓦片包**：离线时地图整块收起、显示一行提示，停靠点文字与「导航」按钮保底。放弃预下载瓦片（每行程 20–60MB、需下载/进度/清理 UI、OSM 条款不鼓励批量抓取）。导航本就靠 Google Maps 提前下载的离线地图（打包清单 p4）。
4. **弱网判定逐件超时，不设全局离线模式**：`navigator.onLine` 在蜂窝弱网下谎报在线，全局横幅治标不治本。地图瓦片 4 秒未到→收起；实时耗时（OSRM）5 秒超时→一行提示；天气改为**永不空白**——上次成功数据一直展示并标注数据时间，过期只触发后台刷新。`navigator.onLine=false` 仅用于跳过等待、直接降级。
5. 公园图鉴（parks.html 一族）同源文件顺手缓存，**不承诺**其离线可用（外部依赖不在保障范围）。

## Consequences

- 部署流程新增约定：**改 sw.js 必须升其内部 CACHE_VERSION**；上线验证除 SHA-1 外增加"断网重载"检查。
- 用户侧要求：出发前至少在线打开一次 App，让缓存装满。iOS 长期不用可能被系统清缓存——出发前打开一次即恢复。
- 回滚手段：部署一个自毁 sw.js（unregister + caches 清空）即可全网撤除。
