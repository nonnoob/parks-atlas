# 口令即图鉴：多口令并存 + 属主侧异常自愈

日期：2026-07-09。背景：云存档上线（ADR-0010）后用户要求：同一设备支持多个口令、每个口令一本独立图鉴、自由切换。随后线上暴露两个后果性 bug（换口令时旧数据泄漏进新图鉴→刷新全标"签名异常"；异常记录只能手动逐条删）。三个决策一起定型了"口令"的最终语义。

## 决策

1. **口令即图鉴，没有账号**。一个口令 ↔ 一本图鉴 ↔ 一份云存档，本地存储按口令派生键分槽。放弃"一设备一身份 + 口令不符即拒绝"的保护模型（v27 曾短暂上线）：那个模型防 typo 但堵死了多本并存。typo 防护改由**新口令确认**承担——入口遇到从未见过的口令，必须确认"开新图鉴"才会创建，防止手滑悄悄多出一本空图鉴。
2. **属主侧异常记录自愈**。验证失败的打卡记录在持有者进入图鉴时自动删除（本地与云端同步清掉），不再要求手动清理。取舍：防篡改的**展示**价值只保留在分享图鉴（只读视图仍标红揭穿伪造）；属主自己的图鉴里，坏记录既不计入进度也无法修复，留着只有噪音。代价：若有人改了属主本地存储，痕迹会被静默清除而非警告——接受，因为能改本地存储的人本来就能整本删除，签名机制防的是"伪造有效记录"，这一点不受影响。
3. **切槽必清态**。账本 load 空槽时完全重置内存状态（身份、记录、密钥缓存），新建身份时清空旧记录——杜绝跨图鉴数据泄漏（v28 的污染事故即源于残留）。

## Consequences

- 口令丢失 = 那本图鉴永久失联（无账号找回路径）；口令泄露 = 对方可打卡、可恢复。帮助文案维持"请保管好口令"。
- 两人恰好同口令 = 同一本图鉴。概率可忽略；后到者会先看到别人的图鉴而非静默覆盖。
- 本地存储槽键与口令派生参数（同 ADR-0010 查找键）**不可变**：各设备已按此分槽。
- 分享图鉴的防伪语义不变：伪造/篡改的分享内容永远标红，不受属主自愈影响。
