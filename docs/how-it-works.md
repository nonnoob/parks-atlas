# 原理笔记 · How it works

面向想读懂这套系统的人。部署步骤见 [self-hosting.md](self-hosting.md)，冻结格式见 [invariants.md](invariants.md)，决策沿革见 [adr/](adr/)。本文只讲机制本身：每一层为什么非这么做不可。

## 1. Service Worker：版本号是缓存的总开关

sw.js 的全部缓存——预缓存清单，加上运行时 stale-while-revalidate 顺手存下的每个响应——都放在 `caches.open(CACHE_VERSION)` 这**一个桶**里。桶名就是版本号字符串。

生命周期决定了换血方式：

- 浏览器每次导航都会拿网络上的 sw.js 和已装的逐字节比对（最长 24 小时也会强制查一次）。**字节不同 = 新 SW**，触发 install。
- install：开新桶（如 `atlas-v2`），装预缓存清单。
- activate：`caches.keys()` 里凡是名字不等于当前版本的桶，**整桶删除**。`skipWaiting()` + `clients.claim()` 让新 SW 跳过等待立即接管所有页面。

由此推出两条铁律：

**改 sw.js 必升 CACHE_VERSION。** 版本不升，字节不变，install/activate 都不跑，老桶永远不清。

**从仓库删文件救不了老访客。** fetch 策略是 stale-while-revalidate：命中缓存先回旧副本，后台 revalidate；拿到 404 时 `if (r && r.ok)` 不成立——既不覆盖也不删除缓存。被删的文件在老访客的桶里**永生**，下次照样命中。唯一的清法就是升版本，让 activate 连桶端掉。

代价直说：升版本 = 全量重新预缓存，老访客多下载一次全套静态资源。所以版本号不白升，攒着改动一起升。

还有一个开发期陷阱：本地调试时 SW 会把你自己也当"老访客"——改完代码刷新看到的是旧文件。要么带新的 `?v=` 参数，要么 DevTools 里 unregister 再刷。

## 2. 云存档为什么要一个 Cloudflare Worker

需求：静态站（GitHub Pages，无后端）把数据写进 GitHub 仓库。写 GitHub Contents API 需要 token。而 token 放不进页面代码，三个独立致命点：

1. 仓库公开，页面代码里的 token 就是公开的；
2. GitHub 扫描公开仓库里自家 token，发现即**分钟级自动吊销**——藏也藏不住，混淆也没用；
3. 泄露的写权限 token = 任何人可写你的仓库。

所以 token 必须待在一个服务端。最小服务端就是一个 Cloudflare Worker（免费档、单文件、无依赖）：token 存 Worker Secret，页面只跟 Worker 说话。Worker 只有两个动作：按查找键读回加密档、验签后代写加密档。仓库指向、分支、目录、允许来源全部走 env 变量（`worker/cloud-worker.js` 顶部 DEFAULTS 兜底），fork 者不改代码只改配置。

Worker 挂了怎么办：前端 `CLOUD` 判空 + 全链路 try/catch，云同步静默降级，App 纯本地照常跑，失败标 `np_dirty` 下次动作重试。云是增强，不是依赖。

## 3. 口令的两条派生路：一把口令，两把钥匙

没有账号系统。口令是唯一秘密，从它派生出两把用途完全相反的钥匙：

**钥匙一：锁私钥（随机盐）。** 每本图鉴生成一对 ECDSA P-256 密钥；私钥用 `PBKDF2(口令, 随机盐 ×150000)` 派生的 AES-GCM 密钥加密后存储。盐是随机的、跟着数据走——同一口令在不同图鉴上派生出不同密钥，预计算字典（彩虹表）无从下手。

**钥匙二：查找键（固定盐 `np-cloud-v1`）。** 云存档的文件名 = `PBKDF2(口令, 固定盐)` 的 128 位十六进制。盐**必须**固定：任何设备、任何时刻，同一口令要能独立算出同一个键——这就是"换个手机输入口令，图鉴自己回来"的全部实现，没有任何服务端账号表。代价也随之而来：固定盐意味着这个键可被字典攻击预计算,所以查找键只用来**定位**密文，不用来解密——解密仍靠钥匙一。

打卡记录的防伪：每条记录是 `sign(私钥, id|日期|备注)`。改任何一个字段验签必失败，显示为"异常记录"。注意边界——**签名防伪造，不防删除**：能碰到你 localStorage 的人可以删记录，但造不出一条验签通过的假记录。属主自己的图鉴里，验签失败的记录进门即清（自愈）；只读分享里则标红示众，这才是防伪的展示场。

云 bundle 是整包加密：`AES-GCM(idn + 全部有效记录)`，外层只露公钥、编号、盐、密文和**对密文的签名**。仓库是公开的，但从仓库里只能看出"有一本图鉴、多大、何时更新"，看不出去过哪座公园。

## 4. TOFU 公钥钉扎：第一个写入者赢

Worker 是公网可 POST 的，怎么防止别人覆盖你的云档？

Trust On First Use：某个查找键**第一次**被写入时，Worker 把 bundle 里的公钥记为该键的主人（钉扎）；此后每次写入,先取已存文件里钉扎的公钥,验新 bundle 的密文签名——不匹配回 409，验签失败回 403。没有你的口令就派生不出你的私钥，就签不出能过验的名，就覆盖不了你的档。Worker 全程不需要知道口令,也解不开内容。

这不是纸面设计。2026-07-10 实战验证过一次：token 还没配好时,持有者误在真口令下"开了新图鉴"（新密钥对，同一查找键），本地新身份往云端推送——Worker 用钉扎的老公钥验签，409 拒收，云端老档毫发无损。删掉本地误建的空槽、重输口令，一切照旧。

## 5. 多端收敛：记录级合并，没有冲突界面

同步策略刻意简单：打卡后防抖 1.2 秒推整包；解锁时拉云档对账。对账是**记录级的**：云端有本地没有的，逐条验签后收进来（`added`）；本地有云端没有的，回传（`extra` 触发一次推送）。记录以公园 id 为键、一园一条,不存在"同一条记录两个版本"的合并难题——这是数据模型换来的免费午餐。

这个机制吞得下真实事故。拆库迁移期间，同一持有者在新旧两个站各开一个标签页，同一秒往两个仓库各写了一份 bundle（iv 不同,密文必不同）。看似分叉，实则无害：两个标签页共享同一 localStorage,本地状态本来就是超集,下次解锁对账时差额自动回传，云端收敛到全集。唯一要做的是尽快关掉旧写入口，别让分叉持续。

## 6. 这套设计防什么、不防什么

| 防 | 不防 |
|---|---|
| 伪造有效打卡记录（签名） | 本地存储被物理删改（能删存储的人本来就能删整本） |
| 无口令者覆盖云档（TOFU） | 口令丢失（无账号即无找回，丢了就是丢了） |
| 仓库/传输泄露游玩记录（整包加密） | 口令泄露（拿到口令 = 拿到图鉴，全部权限） |
| 手滑口令悄悄开新本（"开新图鉴"确认） | 两人恰好同口令（= 同一本图鉴，概率上接受） |

安全模型一句话：**口令即身份，密码学只保证"没有口令的人什么都做不了"，不多也不少。**
